# 第4章 Prompt 工程基础

> 模型再强，提示词写不好也白搭。这是 AI 编程最核心的技能。

## 4.1 为什么 Prompt 如此重要？

同一个模型，不同的提示词，结果天差地别：

```python
# 差的提示词
"写个排序"

# 好的提示词
"用Python实现快速排序，要求：
1. 使用类型注解
2. 处理空列表边界情况
3. 添加docstring说明时间复杂度
4. 包含简单的单元测试"
```

**Prompt 工程 = 学会如何跟 AI 有效沟通。**

## 4.2 提示词的基本结构

一个好的提示词通常包含以下要素：

```python
prompt = """
## 角色 (Role)
你是一个资深Python开发者，擅长写简洁优雅的代码。

## 任务 (Task)
实现一个LRU缓存装饰器。

## 约束 (Constraints)
- 使用functools.lru_cache或自己实现
- 支持maxsize参数
- 添加缓存命中率统计

## 输出格式 (Format)
- 完整的Python代码
- 包含docstring和类型注解
- 附带使用示例
"""
```

### 四要素模型

| 要素 | 作用 | 示例 |
|------|------|------|
| **角色** | 定义 AI 的身份 | "你是一个安全专家" |
| **任务** | 明确要做什么 | "审查以下代码" |
| **约束** | 限制条件 | "只关注SQL注入风险" |
| **格式** | 指定输出形式 | "用Markdown表格列出" |

## 4.3 系统提示词（System Prompt）

系统提示词是「上帝视角」，定义 AI 的全局行为：

```python
# 好的系统提示词
system_prompt = """
你是一个Python代码审查助手。

## 行为规则
1. 只关注代码质量问题，不做功能解释
2. 按严重程度排序：错误 > 性能 > 风格
3. 每个问题给出具体的修复建议
4. 使用简洁的中文，不要啰嗦

## 输出格式
用Markdown列表，每个问题包含：
- 🔴/🟡/🟢 严重程度标识
- 问题描述
- 修复建议（含代码）
"""

messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": f"审查这段代码：\n```python\n{code}\n```"}
]
```

### 系统提示词 vs 用户提示词

```python
# 系统提示词：持久的、定义行为的
{"role": "system", "content": "你是一个翻译助手，只做中英互译"}

# 用户提示词：具体的、一次性的
{"role": "user", "content": "把这段翻译成英文：你好世界"}
```

## 4.4 Few-shot 提示

给 AI 几个示例，让它学会你的格式：

```python
few_shot_prompt = """
## 任务
将用户的口语化描述转换为结构化的JSON。

## 示例

输入: "帮我找北京明天的天气，温度在20度以上"
输出:
```json
{
  "intent": "weather_query",
  "location": "北京",
  "date": "明天",
  "condition": {"temp_min": 20}
}
```

输入: "订一张后天从上海到广州的机票"
输出:
```json
{
  "intent": "book_flight",
  "date": "后天",
  "from": "上海",
  "to": "广州"
}
```

## 现在处理
输入: "查一下本周五深圳有什么好吃的日料"
"""
```

**关键：示例数量 2-5 个最佳，太多浪费 Token，太少效果不好。**

## 4.5 思维链（Chain of Thought）

让 AI 分步骤思考，提高推理准确率：

```python
# 直接提问（容易出错）
"如果一个球从10楼掉下来，经过多少秒到达地面？"

# 思维链提问（更准确）
"""
如果一个球从10楼掉下来，经过多少秒到达地面？

请按以下步骤推理：
1. 确定每层楼的高度
2. 计算总高度
3. 使用自由落体公式计算时间
4. 给出最终答案
"""
```

### 强制思维链

```python
cot_prompt = """
分析这段代码的性能问题。

<thinking>
让我逐步分析：
1. 首先看循环结构...
2. 然后看数据结构选择...
3. 最后看算法复杂度...
</thinking>

基于以上分析，我的结论是...
"""
```

## 4.6 提示词模板化

用 Python 代码管理提示词，而不是拼接字符串：

```python
# code/ch04/prompt_templates.py
from string import Template

# ❌ 不好：硬编码
prompt = f"把这段代码翻译成{lang}：{code}"

# ✅ 好：模板化
class PromptTemplates:
    CODE_REVIEW = Template("""
请审查以下 $language 代码，关注 $focus_area：

```$language
$code
```

输出格式：Markdown 列表，按严重程度排序。
""")

    TRANSLATE = Template("""
将以下 $source_lang 代码翻译为 $target_lang：

```$source_lang
$code
```

要求：
- 保持相同的逻辑和功能
- 使用 $target_lang 的惯用写法
- 保留注释并翻译
""")

# 使用
prompt = PromptTemplates.CODE_REVIEW.substitute(
    language="Python",
    focus_area="性能和安全",
    code=code_snippet
)
```

## 4.7 提示词调试技巧

### 1. 从简单开始，逐步添加

```python
# v1: 最简版本
"写一个排序函数"

# v2: 添加约束
"写一个快速排序函数，使用类型注解"

# v3: 添加格式
"写一个快速排序函数，使用类型注解，包含docstring"

# v4: 添加测试
"写一个快速排序函数，使用类型注解，包含docstring和单元测试"
```

### 2. 负面提示（说不要什么）

```python
prompt = """
实现一个文件上传功能。

## 不要
- 不要使用第三方库
- 不要做过度抽象
- 不要添加不需要的错误处理
- 不要在代码中写TODO
"""
```

### 3. 用 XML 标签分隔内容

```python
prompt = """
请分析以下代码：

<code>
def add(a, b):
    return a + b
</code>

<requirements>
- 处理非数字输入
- 支持浮点数
</requirements>

<output_format>
Python代码 + docstring
</output_format>
"""
```

## 4.8 常见陷阱

| 陷阱 | 问题 | 解决 |
|------|------|------|
| 提示词太模糊 | "写个好代码" | 明确"好"的标准 |
| 信息过载 | 一次性塞太多要求 | 分步骤提问 |
| 缺少示例 | AI 理解偏差 | 加 few-shot 示例 |
| 没有约束 | 输出不稳定 | 明确格式和长度 |
| 忽略系统提示 | 行为不一致 | 用 system prompt 定义角色 |

## 本章小结

- Prompt 工程 = 学会跟 AI 有效沟通
- 四要素：角色 + 任务 + 约束 + 格式
- System Prompt 定义全局行为
- Few-shot 给示例，CoT 让 AI 分步思考
- 模板化管理提示词，而不是硬编码
